## 一、Agent 的本质

很多教程一上来就教你怎么调 API，但忽略了最关键的问题：**Agent 到底是什么？**

一句话定义：Agent 是一个能**自主干活**的 AI 系统——你给它一个目标、一个工具箱和一套边界，它在循环里推进任务，直到完成或停下。

Agent 的核心由四个组件组成：

```
Agent = 大脑(LLM) + 手脚(Tools) + 记忆(Memory) + 主见(Autonomy)
```

在实际生产中，你还需要第五层——**护栏(Guardrails)**。预算、权限、审批、审计、沙箱，缺一不可。没有护栏的 Agent，迟早从"自己干活"变成"自己闯祸"。

### 自主性等级

Agent 不是一个非黑即白的概念，而是一个谱系。按自主性可以分为六个等级：

| 等级 | 名称 | 特征 | 典型例子 |
|------|------|------|---------|
| L0 | Chatbot | 问一句答一句 | ChatGPT 基础对话 |
| L1 | Tool Agent | 调 API 返回结果 | GPTs Actions |
| L2 | ReAct Agent | 思考→行动→观察，循环 | 本文核心 |
| L3 | Planning Agent | 先拆解计划再执行 | 复杂任务编排 |
| L4 | Multi-Agent | 多个 Agent 分工协作 | 企业级系统 |
| L5 | Autonomous | 长期自主运行 | 尚不成熟 |

大多数人在用的不过是 L0-L1。本文的目标是把你的 Agent 推到 **L2-L3**。

## 二、ReAct 循环：Agent 的心脏

ReAct = Reasoning + Acting，是 Agent 最核心的运行机制。搞懂它，你就搞懂了 Agent 怎么"自己干活"。

### 2.1 三阶段循环

```
[Reason 思考]  →  [Act 行动]  →  [Observe 观察]
      ↑________________________________↓
```

- **Reason（思考）**：分析当前情况，决定下一步。只想一步，不要让它想太远。
- **Act（行动）**：调用工具执行。每轮只推进一个关键动作。
- **Observe（观察）**：记录执行结果，客观记录，不做判断。

这解决了 LLM 的两个天生缺陷：**信息过时**（只用训练数据）和**无法验证**（编完就算）。ReAct 强迫 LLM 停下来——先去查，查完再看，然后再决定。

### 2.2 六种终止条件（一定要设！）

ReAct 循环最大的坑是"停不下来"。生产环境必须设置明确的终止条件：

| 条件 | 说明 | 优先级 |
|------|------|--------|
| 用户中断 | 用户主动停止 | 最高 |
| 任务完成 | LLM 明确表示完成 | 高 |
| 预算耗尽 | 达到 Token/成本上限 → 强制停止 | **硬护栏** |
| 超时 | 达到时延上限 → 强制停止 | **硬护栏** |
| 结果收敛 | 连续两次观察结果相似，没有新进展 | 中 |
| 最大轮数 | 达到预设 MaxIterations 兜底 | 兜底 |

关键配置：
- **MaxIterations（10-15）**：防无限循环。Agent 可能卡在同一个搜索词上反复查
- **MinIterations（最少 1）**：防偷懒。有些 Agent 第一轮就说"搞定了"，其实啥工具都没调
- **ObservationWindow**：只保留最近几条观察结果，老的压缩成摘要，控制上下文长度

## 三、工具设计：Agent 的"手脚"

工具让 LLM 从"会说"变成"能做"。但工具不是万能药——description 写烂了，LLM 照样选错工具、填错参数。

### 3.1 工具的完整结构

一个完整的工具由三部分组成：

**① 元数据**：工具叫什么、干什么用、有什么生产约束

```python
class ToolMetadata:
    name: str               # 工具名
    description: str        # 干什么的（给 LLM 看）
    category: str           # 分类：search, monitoring, operation...
    rate_limit: int = 60    # 每分钟最多调几次
    timeout_seconds: int = 30
    dangerous: bool = False # 是否是危险操作
    cost_per_use: float = 0.0
```

这些字段直接影响生产安全。没有 rate_limit，Agent 一分钟调 200 次搜索 API，直接触发 DDoS 保护；没有 dangerous 标记，Agent 可能擅自执行高风险操作。

**② 参数定义**：需要什么输入

```python
class ToolParameter:
    name: str
    type: str  # STRING, INTEGER, BOOLEAN...
    description: str  # 给 LLM 看——决定 LLM 能不能填对
    required: bool = True
    enum: list = None
    min_value / max_value: float = None
```

**description 是最关键的字段**。写法决定 LLM 的填参准确率：

```
❌ "The query"                    → LLM：？？？
✅ "搜索关键词，包含具体实体名和时间范围。
    示例：'payment-service 错误日志 2024'"
```

**③ 执行逻辑**：具体干什么

```python
async def execute(self, **kwargs) -> ToolResult:
    # 1. 限流检查
    if self._is_rate_limited():
        return ToolResult(success=False, error="Rate limited")
    # 2. 参数类型强转（LLM 可能传字符串当整数）
    kwargs = self._coerce_parameters(kwargs)
    # 3. 执行
    try:
        result = await asyncio.wait_for(
            self._execute_impl(**kwargs),
            timeout=self.metadata.timeout_seconds
        )
        return ToolResult(success=True, output=result)
    except asyncio.TimeoutError:
        return ToolResult(success=False, error="执行超时")
    except Exception as e:
        return ToolResult(success=False, error=str(e))
```

注意三层防护：**限流检查 → 参数容错 → 超时控制**。缺一层，生产环境就多一分风险。

### 3.2 如何管理多个工具

真实场景下 Agent 不可能只有一两个工具。当工具超过 5 个时，LLM 的选择准确率会明显下降。

两个关键策略：

1. **按任务类型过滤**：只给 Agent 暴露当前任务需要的 3-5 个工具
2. **功能重复的只留一个**：有 web_search 就别再加 google_search 和 bing_search

### 3.3 ToolResult 的规范

```python
class ToolResult:
    success: bool
    output: Any
    error: str = None
    execution_time_ms: int = None
```

**永远返回 ToolResult**，不要抛异常。结构化的返回让 LLM 能判断"失败了，换个方法重试"还是"成功了，进行下一步"。

## 四、实战：企业级工单处理 Agent

### 4.1 场景与架构

某互联网公司每天收到三类线上工单：

- **告警类**：服务器宕机、延迟飙升
- **查询类**：查日志、查配置
- **操作类**：重启服务、回滚版本

整体流程按 ReAct 模式来设计：

```
[工单输入]
    │
    ▼
┌────────────────────────────┐
│   Phase 1: Reason 思考     │ ← LLM 判断工单类型 + 决定下一步
│   理解意图，选择工具         │
└──────────┬─────────────────┘
           │
    ┌──────┼──────┐
    ▼      ▼      ▼
  告警类  查询类  操作类
    │      │      │
    ▼      ▼      ▼
┌────────────────────────────┐
│   Phase 2: Act 行动        │ ← 调用对应工具
│   查监控 / 查日志 / 做操作  │
└──────────┬─────────────────┘
           │
           ▼
┌────────────────────────────┐
│   Phase 3: Observe 观察    │ ← 记录结果 + 决定下一步
│   是否继续 / 是否完成       │
└──────────┬─────────────────┘
           │
    ┌──────┴──────┐
    ▼              ▼
  继续循环        任务完成
```

### 4.2 工具定义（带生产字段）

```python
from dataclasses import dataclass
from typing import Optional

# --- 工具元数据 ---
RESTART_TOOL = {
    "name": "restart_service",
    "description": "重启指定服务。只有确认问题=服务异常时才调用",
    "parameters": {
        "type": "object",
        "properties": {
            "service": {
                "type": "string",
                "description": "服务名称，如 payment-service"
            },
            "reason": {
                "type": "string",
                "description": "重启原因，必须包含异常证据"
            }
        },
        "required": ["service", "reason"]
    },
    # 生产字段（框架无关，理念通用）
    "dangerous": True,        # 需要用户二次确认
    "timeout_seconds": 60,    # 重启最长等 60s
    "rate_limit": 5,          # 每分钟最多重启 5 次
}
```

### 4.3 带护栏的 Agent 循环

```python
class AgentConfig:
    max_iterations: int = 10      # 防无限循环
    min_iterations: int = 1       # 防偷懒
    token_budget: int = 10000     # Token 预算护栏
    timeout: int = 120            # 总超时

def should_stop(state) -> bool:
    """六种终止条件"""
    # 硬护栏：预算耗尽 / 超时
    if state.tokens >= state.config.token_budget:
        return True
    if time_since_start() >= state.config.timeout:
        return True

    # 最短轮数保护
    if state.turn < state.config.min_iterations:
        return False

    # 任务完成
    if "任务完成" in state.last_thought:
        return True

    # 结果收敛（连续两次观察相似）
    if len(state.observations) >= 2:
        last_two = state.observations[-2:]
        if are_similar(last_two[0], last_two[1]):
            return True

    return state.turn >= state.config.max_iterations

def run_agent(user_input: str):
    messages = [{"role": "system", "content": SYSTEM_PROMPT},
                {"role": "user", "content": user_input}]
    state = AgentState(config=AgentConfig())

    while not should_stop(state):
        # Phase 1: Reason — LLM 思考下一步
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=TOOLS,
        )
        msg = response.choices[0].message
        messages.append(msg)
        state.turn += 1

        if not msg.tool_calls:
            return msg.content  # 任务完成

        # Phase 2: Act — 安全执行工具
        for tc in msg.tool_calls:
            tool = find_tool(tc.function.name)
            if tool.dangerous and not user_confirmed():
                return "高风险操作需用户确认，已暂停"
            result = execute_tool_safely(tool, tc.function.arguments)
            # Phase 3: Observe — 记录观察结果
            messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": str(result)
            })
            state.observations.append(result)
            log_tool_call(tc.function.name, result)

    return f"达到限制（轮次{state.turn}），已升级人工处理"
```

### 4.4 System Prompt

```python
SYSTEM_PROMPT = """你是一个线上工单处理助手。

工单类型与处理方式：
- alert（告警类）：先调用监控工具确认影响范围，再决定恢复方案
- query（查询类）：直接调用查询工具获取数据，整理后返回
- operation（操作类）：高风险操作必须先向用户确认，确认后再执行

核心规则：
1. 你必须使用工具获取信息，不要凭训练数据直接回答
2. 每执行完一个工具，观察结果，再决定下一步
3. 高风险操作（重启、回滚、配置修改）必须用户确认
4. 如果你不确定，先查文档或升级人工
5. 回复要包含具体数据，如错误次数、延迟数值等"""
```

### 4.5 工作流示例

**用户输入**：`"payment-service 延迟飙升到 5s，帮我看一下"`

**ReAct 循环过程**：

| 轮次 | Phase | 内容 |
|------|-------|------|
| 1 | Reason | 判断为 alert 类，需要查日志定位原因 |
| 1 | Act | `query_service_logs(service=payment-service, time_range=5m)` |
| 1 | Observe | 返回 127 次数据库连接超时 |
| 2 | Reason | 数据库连接是根因，检查是否需要重启 |
| 2 | Act | `restart_service(service=payment-service, reason=数据库连接池耗尽)` → 需要用户确认 |
| 2 | Observe | 用户确认后执行，延迟恢复至 <200ms |
| 3 | — | 任务完成，输出报告 |

整个过程 3 轮结束，不到 30 秒。走人工流程至少 10 分钟。

## 五、最佳实践

### 5.1 从最小闭环开始

新手的常见错误是一上来就搞 Multi-Agent。我的建议：

1. **裸调 OpenAI API**，一个工具开始，跑通再扩展
2. 加到 3-5 个工具，按任务类型过滤
3. 加了 Memory（记忆）再考虑框架
4. 需要多 Agent 协作时才上 LangGraph/Shannon

### 5.2 工具设计五大坑

| 坑 | 症状 | 解决 |
|----|------|------|
| description 写太烂 | LLM 选错工具、填错参数 | 写清楚使用场景 + 示例 |
| 不处理失败 | 工具崩了，Agent 也崩 | 永远返回 ToolResult |
| 忘记安全 | 直接 eval 用户输入 | AST 解析 + 白名单 |
| 给太多工具 | LLM 选择困难 | 按任务过滤到 3-5 个 |
| 没有超时 | 一个工具卡死整条链 | metadata.timeout_seconds |

### 5.3 生产必备的监控

```
[Agent] Turn: 1 | Tool: query_service_logs | Params: {service: payment, time_range: 5m}
[Agent] Turn: 1 | Result: success | Tokens: 450 | Latency: 2300ms
[Agent] Turn: 2 | LLM_Thought: "发现数据库连接超时，建议重启"
[Agent] Turn: 2 | Cumulative_Tokens: 3200 | Total_Cost: $0.16
```

结构化日志 + Trace ID 串联，让每轮循环都**可追溯、可回放、可审计**。

## 六、总结

Agent 开发的核心不在于用什么框架、哪个模型，而在于三件事：

1. **ReAct 循环** — 让 LLM 边想边做边看，而不是一口气编答案
2. **工具设计** — description 决定 LLM 能不能用对工具；生产字段（rate_limit、timeout、dangerous）决定能不能上线
3. **护栏体系** — 终止条件、预算控制、权限审查，是 Agent 上生产的前提

送你一句 Wayland 在书里写的话：*"The best way to learn is to build."*

先做一个最小闭环——单 Agent、3-5 个工具、带护栏的 ReAct 循环——跑通再扩展。实际踩坑比读一百篇文章都有用。
