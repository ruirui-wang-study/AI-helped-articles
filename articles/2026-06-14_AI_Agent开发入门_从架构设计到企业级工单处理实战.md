## 一、Agent 的本质

很多教程一上来就教你怎么调 API，但忽略了最关键的问题：**Agent 到底是什么？**

一句话定义：Agent 是一个能**自主干活**的 AI 系统——你给它一个目标、一个工具箱和一套边界，它在循环中推进任务，直到完成或停下。

Agent 由四个组件构成：`Agent = 大脑(LLM) + 手脚(Tools) + 记忆(Memory) + 主见(Autonomy)`。在实际生产中还需第五层——**护栏(Guardrails)**：预算、权限、审批、审计、沙箱，一个都不能少。

**自主性等级**（L0-L5）：大多数人只用到 L0-L1（ChatBot/Tool Agent），本文的目标是 L2-L3（ReAct/Planning Agent）。

---

## 二、ReAct 循环：Agent 的心脏

ReAct = Reasoning + Acting，三阶段循环：

```
[Reason 思考]  →  [Act 行动]  →  [Observe 观察]
      ↑________________________________↓
```

- **Reason**：分析现状，决定下一步。只想一步，不要发散
- **Act**：调用工具执行。每轮一个关键动作
- **Observe**：客观记录结果，不做判断，让下一轮 Reason 去处理

这解决了 LLM 的两大缺陷：**信息过时**（只会用训练数据）和**无法验证**（编完就算）。ReAct 强迫它停下来——先去查，查完再看，再决定。

### 六种终止条件

| 条件 | 说明 |
|------|------|
| 用户中断 | 用户主动停止 |
| 任务完成 | LLM 明确表示完成 |
| 预算耗尽 | Token/成本达上限 → 硬护栏强制停 |
| 超时 | 时延超上限 → 硬护栏强制停 |
| 结果收敛 | 连续两次观察相似，无新进展 |
| 最大轮数 | MaxIterations 兜底 |

关键配置：**MaxIterations(10-15)** 防无限循环，**MinIterations(≥1)** 防Agent第一轮就编答案，**ObservationWindow** 控制上下文长度。

## 三、工具设计：Agent 的"手脚"

### 3.1 工具三部分结构

**① 元数据** — 叫什么、干什么、有什么生产约束：

```python
class ToolMetadata:
    name: str
    description: str                # 给 LLM 看 — 决定会不会选错工具
    category: str                   # 分类：search, monitoring, operation...
    dangerous: bool = False         # 危险操作标记
    rate_limit: int = 60            # 每分钟调用上限
    timeout_seconds: int = 30
```

**② 参数定义** — description 是最关键的字段：

```
❌ "The query"                    → LLM：？？？
✅ "搜索关键词，包含实体名和时间范围。
    示例：'payment-service 错误日志 2024'"
```

**③ 执行逻辑** — 三层防护：限流检查 → 参数类型强转 → 超时控制。

### 3.2 工具管理

工具超过 5 个时，LLM 选择准确率明显下降。两个策略：**按任务类型过滤**（只暴露当前任务所需的 3-5 个）、**功能重复的只留一个**。

### 3.3 ToolResult 规范

永远返回 `ToolResult(success, output, error)`，不抛异常。

## 四、上下文工程与记忆设计

这是新手最容易忽视、但生产环境最关键的环节。

### 4.1 LLM OS 类比：上下文窗口 = RAM

Andrej Karpathy 提出了一个精妙的类比：LLM 就像一个操作系统。

| 传统计算机 | LLM OS | 说明 |
|-----------|--------|------|
| CPU | LLM 模型 | 推理引擎 |
| **RAM** | **上下文窗口** | 有限、易失、昂贵的工作记忆 |
| 硬盘 | 向量数据库 | 持久化存储 |
| 外设 | 工具 | 与外部世界的接口 |

上下文窗口就是 RAM：**有限**（再大也有上限）、**易失**（会话结束就清零）、**昂贵**（每个 token 都要钱）。Agent 系统需要像操作系统管理内存一样，管理上下文里的每一条信息。

### 4.2 Context Engineering（上下文工程）

> Prompt Engineering 关心"怎么说"，Context Engineering 关心"让模型看到什么"。

上下文窗口里有七个争夺空间的成员：System Prompt + 用户请求 + 对话历史 + 长期记忆 + RAG 检索结果 + 工具定义 + 输出格式要求。

**关键现象——Context Rot（上下文腐蚀）**：随着 token 增加，模型准确回忆信息的能力递减。更大的窗口不是万能药——信息检索的准确率随着长度逐渐下滑。

### 4.3 上下文管理四策略

**① Write（写出去）——把信息写到上下文之外**

上下文窗口不够用？别把一切都塞进去。让 Agent 用文件系统做"外置硬盘"：

- **Scratchpad（便签本）模式**：Agent 维护一个 `notes.md`，把任务目标、完成进度、待解问题写进去。需要时再读回来，正好落在上下文末尾（LLM 注意力最强的位置）
- 文件系统 = 无限记忆，上下文 = 工作记忆，两者配合

**② Select（检索回来）——需要时才去取**

核心原则：**即时上下文（Just-in-Time）**。不是把所有信息预加载，而是在需要时才拉取：

- 混合策略：最核心的信息放 System Prompt，具体细节（代码、日志、文档）通过工具按需检索
- **渐进式披露（Progressive Disclosure）**：Agent 像侦探一样，从线索出发逐步拼出全貌

**③ Compress（压缩）——让有限空间装更多**

当对话太长、上下文窗口快满时，触发 **Compaction（压紧）**——用 LLM 把冗长历史压缩成精炼摘要，重新初始化上下文。

Shannon 的三段式保留策略：
- **Primers（前 3 条）**：保留对话开头，用户初始需求和系统设定
- **Summary（中间摘要）**：把几百条消息压成一段语义摘要，保留关键决策和未解决问题
- **Recents（后 20 条）**：保留最近对话，维持连贯性

> 重要：**保留错误上下文**。擦除 Agent 的失败尝试等于擦除了经验——模型看到之前的失败路径，会避免重复犯错。

**④ Isolate（隔离）——拆成多个独立上下文**

当单个上下文窗口装不下整个任务时，拆分为子 Agent。每个子 Agent 在自己干净的窗口中工作，只把精华结果返回主 Agent。

**Token 预算分层管理**：

```python
# 三级预算：Session → Task → Agent
# 三种执行模式：
# - 硬限制：超预算直接拒绝
# - 软限制：超预算发警告，继续执行
# - 审批模式：超预算暂停，等人工确认

# 背压机制——渐进式限流，不断崖式停止
def calc_backpressure(usage_pct: float) -> int:
    if usage_pct >= 0.95: return 1500  # 重度限流
    if usage_pct >= 0.90: return 750
    if usage_pct >= 0.85: return 300
    if usage_pct >= 0.80: return 50
    return 0
```

### 4.4 记忆架构：让 Agent 记住上一次

记忆分为四类：

| 类型 | 时间跨度 | 例子 | 存储方式 |
|------|---------|------|---------|
| 工作记忆 | 秒-分钟 | 正在处理的工单 | 上下文窗口 |
| 会话记忆 | 分钟-小时 | 本次对话历史 | Redis 缓存 |
| 长期记忆 | 天-月 | 用户偏好、工单规律 | 向量数据库 |
| 语义记忆 | 永久 | 常见故障模式 | 向量数据库 |

**三层存储**：PostgreSQL（ACID 事务） + Redis（热数据缓存） + 向量数据库（语义相似度搜索）。

**分层检索**：取 `Recent（最近 5 条）+ Semantic（语义相关 3 条）+ Summary（摘要 2 条）`，去重融合后注入上下文。

```python
# Embedding 把文本转为向量，语义相近的文本向量也相近
# 相似度阈值：0.7 为起点
#   精确场景（如代码搜索）：0.8
#   宽泛场景（如创意探索）：0.6

# MMR 去重 —— 平衡相关性和多样性
# lambda=0.7：偏向相关性（默认）
# lambda=0.5：平衡
# lambda=0.3：偏向多样性
```

### 4.5 工单场景中的应用

回到工单 Agent 的例子，看看上下文和记忆怎么用：

**会话内**：
- 用户排查了 20 轮工单，Agent 日志和诊断数据越来越多
- 用到窗口 75% 时触发压缩：保留前 3 条（初始工单 + 服务名） + 摘要（已排原因、已试方案、未确认的操作） + 最近 5 轮对话
- 不再重复加载已读完的 500 行日志——文件路径留着，需要时再读

**跨会话**：
- 用户第二天回来说"继续昨天的工单"——从向量库中召回语义相关的对话摘要，直接延续上下文
- Agent 记住："payment-service 上周三也出过类似问题，那次是数据库连接池耗尽"
- 积累常见故障模式："延迟飙高 + 数据库超时 → 先查连接池配置"——下次同类工单直接命中

## 五、实战：企业级工单处理 Agent

### 5.1 场景与架构

企业每天收到三类线上工单：**告警类**（宕机、延迟）、**查询类**（查日志、配置）、**操作类**（重启、回滚）。

ReAct 流程 + 上下文管理：

```
[工单输入] → [Reason: LLM 判断类型 + 加载相关记忆]
                  │
           ┌──────┼──────┐
           ▼      ▼      ▼
         告警类  查询类  操作类
                  │
           ┌──────┘
           ▼
     [Act: 调用工具 → Observe: 记录结果 → 压缩/选择性记忆]
                  │
           ┌──────┴──────┐
           ▼              ▼
        继续循环        任务完成 → 保存记忆
```

### 5.2 工具定义（带生产字段）

```python
RESTART_TOOL = {
    "name": "restart_service",
    "description": "重启指定服务。只有定位到服务异常时才调用",
    "parameters": {
        "type": "object",
        "properties": {
            "service": {"type": "string", "description": "服务名"},
            "reason": {"type": "string", "description": "重启原因，须含异常证据"}
        },
        "required": ["service", "reason"]
    },
    "dangerous": True,        # 用户二次确认
    "timeout_seconds": 60,
    "rate_limit": 5,          # 每分钟最多 5 次
}
```

### 5.3 System Prompt + 记忆注入

```python
SYSTEM_PROMPT = """你是一个线上工单处理助手。

工单类型：alert（告警类）/ query（查询类）/ operation（操作类）
规则：
1. 必须使用工具获取信息，不要凭训练数据回答
2. 每执行完一个工具，观察结果，再决定下一步
3. 高风险操作（重启、回滚）必须用户确认
4. 回复包含具体数据（错误次数、延迟数值）
5. 如果该服务上周有类似故障，参考历史经验"""

# 记忆注入：检索相关历史
relevant_memory = memory_retriever.recall(
    query="payment-service 延迟 数据库连接",
    top_k=3
)
if relevant_memory:
    context_block = f"\n【历史相关经验】\n{format_memories(relevant_memory)}"
    messages.append({"role": "system", "content": context_block})
```

### 5.4 带上下文控制的 Agent 循环

```python
class AgentConfig:
    max_iterations: int = 10
    min_iterations: int = 1
    token_budget: int = 10000
    timeout_seconds: int = 120
    # 上下文管理参数
    compress_at_pct: float = 0.75    # 窗口用到 75% 时压缩
    compress_target_pct: float = 0.375
    memory_top_k: int = 3

def run_agent(user_input: str, session_id: str):
    # 1. 加载记忆
    memories = memory_retriever.recall(user_input, top_k=3)
    state = AgentState(session_id=session_id, memories=memories)
    messages = build_initial_context(SYSTEM_PROMPT, user_input, memories)

    while not should_stop(state):
        # Phase 1: Reason
        response = client.chat.completions.create(
            model="gpt-4o", messages=messages, tools=TOOLS
        )
        msg = response.choices[0].message
        messages.append(msg)
        state.turn += 1
        state.tokens_used += response.usage.total_tokens

        if not msg.tool_calls:
            # 任务完成，存记忆
            memory_retriever.store(session_id, user_input, msg.content)
            return msg.content

        # Phase 2: Act
        for tc in msg.tool_calls:
            result = execute_tool_with_safety(tc.function.name, tc.function.arguments)
            messages.append({"role": "tool", "tool_call_id": tc.id, "content": str(result)})

        # Phase 3: 上下文管理 —— 检查是否要压缩
        if state.tokens_used >= state.config.token_budget * state.config.compress_at_pct:
            messages = compress_context(messages, state.config.compress_target_pct)

    return f"已达限制（轮次{state.turn}），升级人工处理"
```

### 5.5 完整工作流

**输入**：`"payment-service 延迟飙升到 5s，帮我看一下"`

| 轮次 | 阶段 | 动作 | 上下文变化 |
|------|------|------|-----------|
| 0 | 初始化 | 检索历史记忆 → 发现上周也出过同类问题 | 注入 1 条相关记忆 |
| 1 | Reason | 判断 alert 类，需查日志 | — |
| 1 | Act | `query_service_logs(service=payment, 5m)` | 返回 127 次超时 |
| 1 | Observe | 数据库连接超时是根因 | — |
| 2 | Reason | 确认需要重启，向用户确认 | — |
| 2 | Act | `restart_service`（用户确认后） | 延迟恢复，错误归零 |
| 2 | 压缩检查 | 本轮 Token 使用未达 75%，不触发 | — |
| 3 | — | 任务完成，输出报告 | 保存记忆："payment-service 数据库连接池耗尽" |

3 轮结束，不到 30 秒。走人工流程至少 10 分钟。

## 六、最佳实践

### 6.1 开发路线

1. **裸调 OpenAI API**，1 个工具跑通 → 3-5 个工具 → 按任务过滤
2. 加入 ReAct 循环 + 终止条件 → 防无限循环
3. 加入上下文压缩 → 控制成本
4. 加入记忆 → 跨会话能力
5. 需要多 Agent 协作 → 再上框架

### 6.2 三层的坑

| 层面 | 常见坑 | 解决 |
|------|--------|------|
| 工具 | description 写太烂 | 写场景 + 示例 |
| 工具 | 直接 eval()` | AST + 白名单 |
| 工具 | 给太多工具 | 过滤到 3-5 个 |
| 上下文 | 不管理上下文 | 按 75% 阈值压缩 |
| 上下文 | 清除错误历史 | 保留失败尝试作为学习信号 |
| 记忆 | 不设相似度阈值 | 0.7 起步，精确场景 0.8 |
| 记忆 | 不设去重 | 95% 阈值 + MMR 重排 |
| 对话 | 不隔离租户 | 返回 NotFound，不泄露 |

### 6.3 监控与可观测性

```
[Agent] Turn: 1 | Tool: query_service_logs | Tokens: 450 | Latency: 2300ms
[Agent] Turn: 2 | Compress: triggered | Before: 8200 → After: 3800 tokens
[Agent] End | Total: 3 turns | Tokens: 12500 | Cost: $0.16
```

**Token 消耗**和**压缩触发**是生产中两个最该盯的指标。一个跑 5 轮的工单可能吃掉 5000-10000 Tokens，不限就等着月底账单吧。

## 七、总结

Agent 开发的核心不在于用什么框架、哪个模型，而在于四件事：

1. **ReAct 循环** — 让 LLM 边想边做边看，而不是一口气编答案
2. **工具设计** — description 决定 LLM 能不能用对工具；生产字段决定能不能上线
3. **上下文工程** — 有限窗口里的精打细算（Write/Select/Compress/Isolate）
4. **记忆系统** — 让 Agent 从"每次都像陌生人"变成"越用越懂你"

送你书里的一句话：*"The best way to learn is to build."*

先做一个最小闭环——单 Agent、3-5 个工具、带护栏的 ReAct 循环——跑通再扩展。实际踩坑比读一百篇文章都有用。
