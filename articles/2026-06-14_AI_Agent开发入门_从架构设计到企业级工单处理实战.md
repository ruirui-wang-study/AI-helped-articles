# AI Agent 开发入门：从架构设计到企业级工单处理实战

> 适合读者：有编程基础、想系统了解 Agent 开发的开发者
> 本文约 2800 字，阅读时间约 8 分钟

---

## 一、引言

2025 年以来，AI Agent 从一个热词变成了实打实落地的东西。无论是代码助手、客服机器人还是自动化工作流，Agent 的核心模式都大同小异：**LLM 做大脑，工具当手脚**。

但很多新手开发者上手时，容易掉进两个坑：

1. **Prompt 万能论** — 觉得写好 prompt 就完事了，忽略了架构设计
2. **过度抽象** — 一上来就搞复杂的 Multi-Agent 框架，结果连单 Agent 都没调通

这篇文章不讲花哨的概念，聚焦**一个企业级工单处理 Agent** 的实战过程，带你走一遍从架构设计到代码实现的完整路径。

## 二、核心架构设计

### 2.1 Agent 的基本运行模式

无论用什么框架，Agent 的本质是一个循环：

```
接收输入 → 理解意图 → 选择工具 → 执行工具 → 处理结果 → 决定下一步
         ↑_____________________________________________________↓
```

这个循环可以跑一次（简单问答），也可以跑多次（复杂任务链）。

### 2.2 三层架构模型

实际项目中，我推荐按三层来组织：

```
┌─────────────────────────────────────┐
│        Orchestration Layer          │
│   (任务编排、状态管理、会话控制)       │
├─────────────────────────────────────┤
│           Tool & Plugin Layer       │
│   (API 调用、数据库查询、文件读写等)    │
├─────────────────────────────────────┤
│            LLM Core                 │
│   (系统 Prompt、工具描述、模型路由)     │
└─────────────────────────────────────┘
```

- **LLM Core**：负责理解意图、拆解任务。核心是写好 System Prompt 和 Tool Definition。
- **Tool & Plugin Layer**：Agent 的"手脚"。每个工具要有清晰的输入输出约定。
- **Orchestration Layer**：管理多轮对话、记忆、任务状态。企业级场景下这层最容易被低估。

### 2.3 企业级的额外考量

相比玩具级 demo，企业级 Agent 需要在四个维度上做得更扎实：

| 维度 | 说明 | 常见做法 |
|------|------|---------|
| **可观测性** | 每个环节的输入输出都要有日志，方便排查 | 结构化 JSON 日志 + Trace ID 串联 |
| **容错机制** | 工具调用失败要有重试和降级策略 | 指数退避重试 + 兜底回复 |
| **安全边界** | 工具权限最小化，防止 Prompt 注入 | 参数白名单 + 敏感信息脱敏 |
| **性能** | 响应速度不能太慢 | 流式输出 + 并行工具调用 + 超时控制 |

很多 demo 跑得挺好，一上生产就崩，问题大多出在这四个维度没做够。

## 三、标准开发流程

实战之前，先明确一个标准的开发流程：

1. **需求定义**：Agent 要解决什么问题？边界在哪？
2. **工具清单**：需要哪些外部能力？API / 数据库 / 文件系统？
3. **Prompt 设计**：System Prompt + 工具描述，这是最需要反复迭代的一步
4. **核心逻辑实现**：搭建 Agent 循环，调用 LLM + Tool
5. **测试与调优**：边界 case、异常场景、幻觉控制
6. **部署与监控**：日志、告警、反馈收集

> **一个经验**：Prompt 迭代花的精力至少占整个开发周期的 40%。别想着一次写好。System Prompt 的每句话都可能影响 Agent 的行为，改一个字都要重新测试一轮。

另一个容易被忽略的步骤是 **边界条件梳理**：LLM 传了非法参数怎么办？工具超时了怎么办？用户问的问题超出 Agent 能力怎么办？这些在需求阶段就要想清楚，而不是等项目上线了才被动发现。

## 四、实战：企业级工单处理 Agent

### 4.1 场景描述

某互联网公司每天收到大量线上工单，分三类：

- **告警类**：服务器宕机、高延迟、错误率飙升
- **查询类**：查日志、查配置、查灰度比例
- **操作类**：重启服务、回滚版本、调整配置参数

目标是开发一个 Agent，能自动分类工单、执行查询操作、在必要时升级给人工处理。

### 4.2 整体架构设计

Agent 的整体流程分为三个步骤：**意图识别 → 分类路由 → 工具执行**。每个步骤都是独立模块，方便后续扩展和替换。

三类工单的处理逻辑如下：

- **告警类**：先查监控确认影响面，再尝试自动恢复（如重启），如果恢复失败则升级人工
- **查询类**：直接查对应系统返回结果，天然安全
- **操作类**：最敏感的一类。需要验证操作人权限、让用户二次确认、执行后验证结果

```
[工单输入]
    │
    ▼
┌──────────────────┐
│   Intent Router   │ ← LLM 判断工单类型
└──────┬───────────┘
       │
       ├──→ 告警类 ──→ [告警诊断 Agent] ──→ 查监控系统 → 执行恢复 → 发送报告
       ├──→ 查询类 ──→ [信息查询 Agent] ──→ 查日志/配置 → 返回结果
       └──→ 操作类 ──→ [操作审批 Agent] ──→ 验证权限 → 确认执行 → 回写结果
                                               │
                                               ▼
                                         升级人工（高风险操作）
```

### 4.3 关键代码示例

以下用 **OpenAI Function Calling** 风格实现，框架无关，方便你迁移到 LangChain / Vercel AI SDK 等框架。

**工具定义：查询服务日志**

```python
tools = [
    {
        "type": "function",
        "function": {
            "name": "query_service_logs",
            "description": "查询指定服务的最近日志，支持按时间范围和关键词过滤",
            "parameters": {
                "type": "object",
                "properties": {
                    "service": {"type": "string", "description": "服务名称"},
                    "time_range": {"type": "string", "description": "时间范围，如 '5m' '1h'"},
                    "keyword": {"type": "string", "description": "关键词过滤，可选"}
                },
                "required": ["service", "time_range"]
            }
        }
    },
    {
        "type": "function",
        "function": {
            "name": "restart_service",
            "description": "重启指定服务，需要审批确认",
            "parameters": {
                "type": "object",
                "properties": {
                    "service": {"type": "string", "description": "服务名称"},
                    "reason": {"type": "string", "description": "重启原因"}
                },
                "required": ["service", "reason"]
            }
        }
    }
]
```

**核心 Agent 循环（简化版）**

```python
def run_agent(user_input: str, messages: list = None):
    if messages is None:
        messages = [{"role": "system", "content": SYSTEM_PROMPT}]
    messages.append({"role": "user", "content": user_input})

    max_turns = 5
    for _ in range(max_turns):
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=tools,
            tool_choice="auto"
        )
        msg = response.choices[0].message
        messages.append(msg)

        if not msg.tool_calls:
            # LLM 决定不再调用工具，返回最终回复
            return msg.content

        for tc in msg.tool_calls:
            result = execute_tool(tc.function.name, tc.function.arguments)
            messages.append({
                "role": "tool",
                "tool_call_id": tc.id,
                "content": str(result)
            })

    return "已达到最大轮次，升级人工处理"
```

**System Prompt（核心部分）**

```python
SYSTEM_PROMPT = """你是一个线上工单处理助手。你的职责是：

1. 理解工单内容，判断类型：告警(alert) / 查询(query) / 操作(operation)
2. 根据类型调用对应工具处理
3. 无法处理时清晰告知，引导升级人工

规则：
- 告警类：先查监控确认影响范围，再执行恢复操作
- 查询类：直接调用查询工具返回结果
- 操作类：高风险操作（重启、回滚）必须二次确认
- 任何时候不要直接执行破坏性操作，需要用户确认后才执行
- 回复要简洁、专业，给出关键数据和操作结果
- 如果你不确定某个操作的影响，先查文档或升级人工"""
```

这个 Prompt 做了几件关键的事：

- **定义角色和边界** — Agent 知道自己是谁、能做什么、不能做什么
- **给出分类逻辑** — 明确三种工单类型的处理方式，降低 LLM 自由发挥的空间
- **预设安全规则** — 高风险操作必须二次确认、不确定就升级人工
- **规范输出格式** — 要求回复简洁专业、包含数据依据，避免 Agent 含糊其辞

**输入工单**：
> "payment-service 延迟飙升到 5s，帮我看一下"

**Agent 执行过程**：
1. 判断为告警类工单，LLM 选择 `query_service_logs` 工具
2. 调用 `query_service_logs(payment-service, 5m)` → 发现大量数据库连接超时报错
3. LLM 分析后回复："payment-service 近 5 分钟出现数据库连接超时（共 127 次），已确认影响范围。建议检查数据库连接池配置。是否需要我重启服务？"
4. 用户确认后调用 `restart_service(payment-service, 数据库连接池耗尽)`
5. 监控数据恢复后，LLM 汇总结果："服务已重启，延迟已恢复至 <200ms，错误率归零"

整个过程 Agent 跑了 3 轮（意图识别 → 查询 → 操作），不到 30 秒完成，如果走人工流程至少需要 10 分钟。

## 五、最佳实践

### 5.1 错误处理

```python
def execute_tool(name: str, args_json: str):
    try:
        args = json.loads(args_json)
        # 参数校验
        if name == "restart_service" and "service" not in args:
            return {"error": "缺少必要参数: service"}
        # 执行
        return real_execute(name, args)
    except json.JSONDecodeError:
        return {"error": "参数解析失败"}
    except Exception as e:
        return {"error": f"工具执行异常: {str(e)}"}
```

### 5.2 日志与可观测性

每轮 Agent 循环都要记录：

```
[Agent] Turn 1 → Tool: query_service_logs, Args: {service: payment, time_range: 5m}
[Agent] Turn 1 ← Result: {status: "timeout_errors", count: 127}
[Agent] Turn 2 → LLM Response: "已检测到数据库连接超时..."
```

用结构化日志（JSON 格式），每行一条，串联 Trace ID，方便后续做离线分析和问题回放。生产环境建议把日志打到 ELK 或 Loki 这类集中式日志平台。

另一个容易被忽视的点是 **Token 消耗监控**。Agent 多轮循环的 Token 消耗是线性增长的，一个跑 5 轮的工单可能吃掉 5000~10000 Tokens。如果不做监控，月底账单会吓你一跳。

### 5.3 安全边界

- **工具权限分级**：只读/读写/高风险，每个工具单独标注
- **操作确认**：高风险操作必须用户二次确认
- **敏感信息过滤**：工具返回的日志中可能包含密码、Token，输出前脱敏
- **限流与超时**：每个工具调用设超时，防止 Agent 被卡死

## 六、总结

Agent 开发并不神秘。它不像传统后端开发那样有一套固定范式，也不像纯 Prompt 工程那么简单。它的挑战在于**把 LLM 的不确定性关进笼子里**——通过架构设计、工具约束、Prompt 规范，让 AI 在可控范围内发挥最大价值。

回顾全文，核心就三件事：

1. **设计好工具层** — 工具就是 Agent 的能力边界
2. **写好 Prompt** — System Prompt 决定了 Agent 的行为质量
3. **管好循环** — 轮次控制、错误处理、日志记录

如果你正在规划自己的第一个 Agent 项目，建议：

- 先做一个**单 Agent + 3~5 个工具**的最小闭环
- 跑通后再加 Memory、Multi-Agent 等复杂能力
- 始终从**业务场景**出发，不要为了用 Agent 而用 Agent

工单处理只是一个起点。把这个模式吃透，换个场景——代码审查、客服、数据分析——逻辑都一样。关键不在于用什么框架、哪个模型，而在于**把业务逻辑清晰地映射到 Agent 的工具有限集和 Prompt 规范里**。

关于框架选择，我的建议是：初期不需要上 LangChain 这样的大框架。先裸调 OpenAI API 把逻辑跑通，它让你对 Agent 的每一轮循环都有完全的控制。等业务复杂度上升到需要 Memory、RAG、Multi-Agent 协作时，再引入框架来减少样板代码。

希望你读完这篇文章后，能动手写一个属于自己的 Agent。从最简单的工具调用开始，跑通后再逐步加功能。实际踩坑比看一百篇文章都有用。
