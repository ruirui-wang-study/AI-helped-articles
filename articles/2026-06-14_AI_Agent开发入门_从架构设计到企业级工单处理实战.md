## 一、Agent 的本质与模式概览

### 1.1 什么是 Agent

Agent = 一个能**自主干活**的 AI 系统。你给它一个目标、一个工具箱和一套边界，它在循环中推进任务，直到完成或停下。核心组件：`大脑(LLM) + 手脚(Tools) + 记忆(Memory) + 主见(Autonomy) + 护栏(Guardrails)`。

### 1.2 单 Agent 的四种核心模式

本文设计的 Agent 采用**单一 Agent**（非 Multi-Agent），但集成了四种模式：

| 模式 | 等级 | 做什么 | 本文用法 |
|------|------|--------|---------|
| **ReAct** | L2 | 思考→行动→观察循环 | 主体骨架，驱动每轮交互 |
| **Chain-of-Thought** | L2+ | 显式的逐步推理 | 嵌入 Reason 阶段，让思路可审计 |
| **Planning** | L3 | 复杂任务先分解再执行 | 多步骤工单启动规划 |
| **Reflection** | L3+ | 输出前自我评估，不达标重试 | 高风险操作/报告类工单启用 |

它们的关系：**ReAct 是骨架，CoT 是推理方式，Planning 和 Reflection 是可选增强模块**，按任务复杂度动态启用。

### 1.3 不是 Multi-Agent

文章里的架构图虽然画了"告警诊断 Agent"、"信息查询 Agent"等分支，但它们**不是独立的 Agent**，而是同一个 ReAct 循环内的**路由分支**。为什么保持单 Agent？

- 工单场景的任务链短（大多 2-5 轮），Multi-Agent 的编排开销得不偿失
- 单 Agent 的上下文连贯，不需要跨 Agent 传递状态
- 等业务复杂度到需要专门 Agent 监控数据库、Agent 执行变更时，再拆不迟

## 二、ReAct 循环 + Chain-of-Thought：Agent 的骨架与推理

### 2.1 三阶段循环（ReAct）

```
[Reason 思考 → CoT 逐步推理]  →  [Act 行动]  →  [Observe 观察]
         ↑___________________________________________↓
```

每个 Reason 阶段不再是一个模糊的"思考"，而是显式的 Chain-of-Thought 推理：

```
【Reason 阶段 - CoT 推理过程】
→ 步骤1: 工单描述是"payment-service 延迟飙升到 5s"
→ 步骤2: 这是 alert 类型，需要先查日志确认根因
→ 步骤3: 查日志工具可过滤最近 5 分钟、关键词"timeout"
→ 步骤4: 查完日志后，根据结果决定是否需要重启
→ 因此: 下一步调用 query_service_logs(service=payment, time_range=5m)
```

这让 Agent 的每一步思考**可审计、可回溯**。出错了能定位到具体推理步骤。

### 2.2 六种终止条件

| 条件 | 说明 |
|------|------|
| 用户中断 | 用户主动停止 |
| 任务完成 | Agent 明确表示完成 |
| 预算耗尽 | Token/成本达上限 → 硬护栏 |
| 超时 | 时延超上限 → 硬护栏 |
| 结果收敛 | 连续两次观察相似，无新进展 |
| 最大轮数 | MaxIterations 兜底 |

关键配置：**MaxIterations(10-15)** 防无限循环，**MinIterations(≥1)** 防偷懒，**ObservationWindow** 控制上下文。

## 三、工具设计：Agent 的"手脚"

### 3.1 工具三部分

**① 元数据**：

```python
class ToolMetadata:
    name: str
    description: str          # 给 LLM 看 — 决定会不会选错
    category: str             # search, monitoring, operation...
    dangerous: bool = False
    rate_limit: int = 60
    timeout_seconds: int = 30
```

**② 参数定义** — description 决定 LLM 填参准确率：

```
❌ "The query"                → LLM：填什么？
✅ "服务名，如 payment-service、user-service"
```

**③ 执行逻辑** — 三层防护：限流检查 → 参数类型强转 → 超时控制。

### 3.2 工具管理

工具超过 5 个时 LLM 准确率下降。策略：**按任务类型过滤**和**功能重复只留一个**。

## 四、上下文工程与记忆系统

### 4.1 LLM OS 类比

上下文窗口 = LLM 的 RAM：有限、易失、昂贵。Agent 系统像 OS 管理内存一样管理上下文。

### 4.2 上下文管理四策略

| 策略 | 做法 | 工单场景应用 |
|------|------|-------------|
| **Write** | 写出去 | Agent 维护 notes.md，记录排查进度和已排除的原因 |
| **Select** | 检索回来 | 按需拉取日志、监控数据，而不是预加载所有 |
| **Compress** | 压缩 | 窗口用到 75% 时触发压缩，保留前 3 条+摘要+最近 20 条 |
| **Isolate** | 隔离 | 复杂工单拆成子排查单元，各自独立上下文 |

### 4.3 记忆系统

四类记忆 × 三层存储（PostgreSQL + Redis + 向量库）：

- **会话内**：压缩历史 + 保留错误上下文（失败尝试是学习信号）
- **跨会话**：分层检索（Recent + Semantic + Summary），去重融合后注入上下文
- **Agent 记住**："payment-service 上周也出过数据库连接池问题"——下次同类工单直接命中

## 五、核心设计：四模式融合的 Agent 循环

这是本文的核心——一个融合了 ReAct、CoT、Planning、Reflection 的完整 Agent 循环。

### 5.1 整体流程

每个工单的处理分为三个阶段：

```
[工单输入]
    │
    ▼
┌─────────────────────────────────────┐
│  Phase 1: 复杂度评估 + 规划决策      │
│  ─ LLM 判断工单复杂度               │
│  ─ 复杂工单 → 启动 Planning         │
│  ─ 简单工单 → 直接进入 ReAct        │
└──────────────┬──────────────────────┘
               │
    ┌──────────┴──────────┐
    ▼                      ▼
  简单工单                复杂工单
    │                      │
    ▼                      ▼
┌──────────────────────────────────────┐
│  Phase 2: ReAct + CoT 循环           │
│  ─ Reason 阶段使用 CoT 逐步推理       │
│  ─ Act 阶段调用工具                   │
│  ─ Observe 阶段记录结果并检查压缩       │
│  ─ 达到终止条件后退出                  │
└──────────────────┬───────────────────┘
                   │
                   ▼
┌──────────────────────────────────────┐
│  Phase 3: Reflection 质量评估         │
│  ─ LLM 评估输出质量（覆盖率/准确性）   │
│  ─ 高风险操作 + 低质量 → 带反馈重生成  │
│  ─ 达标 → 输出结果 + 存入记忆          │
└──────────────────────────────────────┘
```

### 5.2 配置层

```python
@dataclass
class AgentConfig:
    # ReAct 参数
    max_iterations: int = 10
    min_iterations: int = 1
    token_budget: int = 10000
    timeout_seconds: int = 120
    observation_window: int = 5  # 保留最近观察数

    # CoT 参数
    cot_enabled: bool = True
    cot_max_steps: int = 5       # 推理步骤上限

    # Planning 参数
    planning_trigger_score: float = 0.5  # 复杂度超此值启动规划
    planning_max_subtasks: int = 7

    # Reflection 参数
    reflection_enabled: bool = True
    reflection_max_retries: int = 1
    reflection_threshold: float = 0.7

    # 上下文管理
    compress_at_pct: float = 0.75
    compress_target_pct: float = 0.375
    memory_top_k: int = 3
```

所有参数都设默认值，让 Agent 能直接跑起来。新手可以先不管这些参数，跑通了再按需调整。

### 5.3 核心循环实现

```python
def run_agent(user_input: str, session_id: str):
    config = AgentConfig()
    state = AgentState(session_id=session_id)

    # Phase 0: 注入相关记忆
    memories = memory_retriever.recall(user_input, top_k=config.memory_top_k)
    messages = build_initial_context(SYSTEM_PROMPT, user_input, memories)

    # Phase 1: 复杂度评估 + 是否启动规划
    complexity = assess_complexity(user_input)

    if complexity > config.planning_trigger_score:
        # 启动 Planning：分解任务，按 DAG 拓扑排序执行
        plan = decompose_task(user_input, config.planning_max_subtasks)
        for subtask in topological_sort(plan.subtasks):
            result = execute_subtask(subtask)
            state.accumulate(result)
            if len(state.observations) > config.observation_window:
                state.observations = state.observations[-config.observation_window:]
        synthesis = synthesize_results(state, user_input)
        messages.append({"role": "assistant", "content": f"规划概要：{synthesis}"})

    # Phase 2: ReAct + CoT 循环
    while not should_stop(state, config):
        # Reason 阶段：用 CoT 显式推理
        response = client.chat.completions.create(
            model="gpt-4o",
            messages=messages,
            tools=TOOLS,
            # CoT 引导：让 LLM 输出逐步推理
            extra_body={
                "prompt_template": build_cot_reason_prompt(user_input, config.cot_max_steps)
            } if config.cot_enabled else None
        )
        msg = response.choices[0].message
        messages.append(msg)
        state.turn += 1

        if not msg.tool_calls:
            break  # 任务完成

        # Act 阶段
        for tc in msg.tool_calls:
            result = execute_tool_safely(tc.function.name, tc.function.arguments)
            messages.append({"role": "tool", "tool_call_id": tc.id, "content": str(result)})
            state.observations.append(result)

        # 上下文管理：检查是否要压缩
        if state.tokens_used >= config.token_budget * config.compress_at_pct:
            messages = compress_context(messages, config.compress_target_pct)

    final_answer = parse_final_answer(messages)

    # Phase 3: Reflection 质量评估（高风险或报告类工单）
    if config.reflection_enabled and is_high_value_output(user_input):
        result = reflect_on_result(user_input, final_answer, config.reflection_threshold)
        if result.score < config.reflection_threshold:
            # 带反馈重生成一次
            feedback = result.feedback
            messages.append({
                "role": "system",
                "content": f"上一版回复需改进：{feedback}。请根据反馈优化。"
            })
            improved = call_llm(messages)
            final_answer = improved if improved else final_answer

    # 保存记忆
    memory_retriever.store(session_id, user_input, final_answer)
    return final_answer


def should_stop(state, config):
    if state.tokens >= config.token_budget:       return True  # 硬护栏
    if time_since_start() >= config.timeout_seconds: return True
    if state.turn < config.min_iterations:        return False
    if "任务完成" in state.last_thought:           return True
    if observations_converged(state.observations):  return True
    return state.turn >= config.max_iterations
```

### 5.4 CoT 推理 Prompt

```python
def build_cot_reason_prompt(context: str, max_steps: int) -> str:
    return f"""请基于当前情景，逐步推理下一步行动。

情景：{context}

请用以下格式推理：
→ 步骤1：[你的第一个推理]
→ 步骤2：[基于步骤1的进一步推理]
...
→ 因此：[你的结论和下一步行动]

限制：最多 {max_steps} 步推理。"""
```

### 5.5 Reflection 质量评估

```python
def reflect_on_result(query: str, answer: str, threshold: float) -> EvaluationResult:
    prompt = f"""评估以下回答的质量。

用户问题: {query}

回答: {answer}

评估维度（每项 0-1 分）：
1. completeness - 是否覆盖了问题的所有方面
2. correctness - 信息是否准确
3. clarity - 表达是否清楚

总分低于 {threshold} 时，请给出具体改进建议。

输出 JSON:
{{
    "score": 0.75,
    "feedback": "回答缺少...",
    "should_improve": true
}}"""
    result = call_llm(prompt, output_format="json")
    return EvaluationResult(**result)
```

## 六、System Prompt（完整版）

```python
SYSTEM_PROMPT = """你是一个线上工单处理助手。

【工单类型】
- alert（告警类）：查监控 → 确认影响 → 恢复操作
- query（查询类）：查日志/配置 → 返回数据
- operation（操作类）：验证权限 → 用户确认 → 执行

【推理规则】
- 每个 Reason 阶段必须逐步推理，用 → 标记步骤
- 涉及数据时标注来源，例如：日志显示 error_count=127（来自 query_service_logs）
- 区分"事实"和"推测"：有数据支撑的写事实，不确定的写明"推测"

【安全规则】
- 高风险操作（重启、回滚、配置修改）必须用户确认
- 不确定时先查文档或升级人工

【质量要求】
- 回复包含具体数据，不模糊
- 团队积累了若干历史故障模式，在上下文的【历史相关经验】块里，必要时参考
"""
```

## 七、完整工作流演示

### 7.1 简单工单（走 ReAct + CoT）

**输入**：`"payment-service 延迟飙升到 5s，帮我看一下"`

| 轮 | 阶段 | 内容 |
|----|------|------|
| 0 | 记忆注入 | 检索到上周同类故障："payment-service 数据库连接池耗尽" |
| 1 | **Reason + CoT** | → 工单是 alert 类 → 延迟 5s 高于正常值 → 先查日志确认根因 → 调用 query_service_logs |
| 1 | Act | 查日志：127 次数据库连接超时 |
| 1 | Observe | 记录结果，压缩检查（Token 使用未达 75%，不触发） |
| 2 | **Reason + CoT** | → 数据库超时是根因 → 结合历史经验（上周同样问题）→ 建议重启 → 向用户确认 |
| 2 | Act | 用户确认 → 执行 restart_service |
| 2 | Observe | 延迟恢复至 <200ms，错误归零 |
| 3 | **Reflection** | 评估输出：覆盖率 90%，准确率 95% → 达标，输出报告 |
| — | 记忆存储 | 保存："payment-service 数据库连接池耗尽 → 重启恢复" |

3 轮 + 1 次 Reflection，不到 35 秒。

### 7.2 复杂工单（启动 Planning）

**输入**：`"昨天下午 payment-service 间歇性 502，排查原因并给修复方案"`

| 阶段 | 内容 |
|------|------|
| **Phase 1** | LLM 评估复杂度 0:72 → 超 0.5 阈值，启动 Planning |
| **规划分解** | ① 查 14-16 点错误日志 → ② 查同期数据库/Redis 监控 → ③ 查当天上线变更记录 → ④ 综合分析 |
| **拓扑排序** | ①∥②∥③（并行） → ④（依赖前三者） |
| **ReAct 执行** | 每个子任务独立跑自己的 ReAct 循环 |
| **Phase 3** | Reflection 评估：覆盖 85%，还差根因验证 → 补充查连接池配置 → 重评估达标 |
| **输出** | 报告 + 修复步骤 |

## 八、最佳实践

### 8.1 开发路线

1. 裸调 API，1 个工具开始 → 加到 3-5 个
2. 加入 ReAct + 终止条件 → 防无限循环
3. 加入 CoT 到 Reason 阶段 → 推理可审计
4. 加入上下文压缩 → 控制 Token
5. 加入记忆 → 跨会话能力
6. **可选**：Planning（多步排查）+ Reflection（高价值输出）

### 8.2 各模式的使用场景对照

| 模式 | 什么时候用 | 什么时候不用 |
|------|-----------|-------------|
| ReAct | 任何交互场景——它是基础骨架 | 不用 ReAct 就不是 Agent 了 |
| CoT | 需要逻辑推理、多步计算、调试 | 简单问答、事实查询 |
| Planning | 多维度排查、研究报告、跨系统分析 | 单步任务（查个日志、重启服务） |
| Reflection | 操作类工单的最终输出、报告生成 | 简单查询（查完直接给结果） |

## 九、总结

这篇文章设计的 Agent 是**单一 Agent**，但在 ReAct 骨架之上集成了三种增强模式：

- **ReAct**：驱动每轮交互的"思考→行动→观察"循环
- **Chain-of-Thought**：让 Reason 阶段的推理过程外显化，可审计
- **Planning**：复杂工单先拆解后执行，按 DAG 拓扑排序
- **Reflection**：高价值输出前做质量评估，不达标则带反馈重试

四者的关系：**ReAct 是发动机，CoT 是仪表盘，Planning 是导航系统，Reflection 是质检员**。它们不是互斥的，是按需叠加的。
